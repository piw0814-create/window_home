인덱스 - 값을 빠르게 찾는 색인

    index scan : 값 -> 위치 색인에서 몇번만에 점프 (루트 -> 리프 ...)
    but 인덱스가 있다고해서 무조건 쓰는건 아님, DBSM 자체가 SQL문장 보고 통계적으로 효율적으로 작동
    트레이드 오프 : 읽기는 빨라지지만 데이터 변경 시 인덱스도 갱신 -> 쓰기느려짐 + 공간추가

    B+tree 구조 : 인덱스 구현 : 루프-내부노드(어디로 내려갈지 키,포인터), 리프(실제 키+데이터위치(TID)) (리프에도 실제데이터가 있는 건 아님.)
    tid(튜플 식별자) : 테이블에서 행의 물리적 위치(블록번호,블록내 순번(슬롯)) 인덱스 리프는 키와 한꼐 tid를 저장해, 실제 행이 있는 heap(테이블 본체)로 바로 점프

        팬아웃 : 노드 하나가 갈라지는 자식가지수 = 클수록 높이가 낮아짐 -> 내가 원하는 데이터를 찾아가는 시간이 짧아짐
        트리높이 : 루트->리프 단계 수, 값하나 찾는데 필요한 읽기 횟수와 같다
        O(log n) : 데이터가 n배 늘어도 탐색단계는 로그만큼만 증가 = 거의 안늘어남

        등치 검색 : 한점 찾기
        범위 검색 : 시작값 리프 찾고, 리프 연결 리스트 따라 옆으로, 끝값 넘으면 종료

        삽입과 노드 분할 : 꽉찬(키) 리프 에 새 키가 들어오면(오버플로), 둘로 분할해서 가운데 키를 위로 부모로 올려 분할
        *노드는 위치정보를 담는 인덱스 키를 담는 그릇
        무작위 키(UUID) : 삽입이 여기 저기 흩어짐 곳곳에서 분할 -> 쓰기 느림, 단편화
        순차 키(bigserial) : 항상 맨 끝에 추가 -> 분할 적음 -> 쓰기 효율적
        정의는 설계 단계(DDL)에서 PK타입 지정, 이후 insert 마다 설계자가 지정한 PK타입으로 키 부여

        B-tree : 값이 모든 노드에 있음, 무거움, 팬아웃 작고 높이 증가, 범위검색 : 값이 흩어져 트리를 위아래로 순회해야함
        B+tree : 낮은 높이-> 빠른 탐색, 균형 트리-> 일정 성능, 리프 연결->범위 정렬 강함
            삽입 시 분할 비용, DML마다 갱신, 저장공간 추가

    인덱스의 비용 : 쓰기 오버헤드(DML마다 관련 인덱스 모두 갱신)(여러 컬럼에데해 인덱스가 다 부여된 상태이면, 삽입 1회당 인덱스 다 갱신해야함)
        저장공간, 옵티마이저 부담(후보 계획 증가, 잘못된(쓸모없는) 인덱스 선택가능)
        
        많을 수록 좋은게 아니다 -> 자주쓰는 조회 패턴에 맞춰 필요한 만큼만.(많이 만들수록 검색은 빨라져도 입력 수정은 느려짐)

    힙은 들어온 순서대로 쌓임(값 순서x), 인덱스가 준  TID로만 찾을 수있음

    클러스터형 인덱스 : 데이터 자체를 키 순서로 정렬해 저장 -> 힙없이 바로 도달

        힙 접근 불필요 -> 키 조회 매우 빠름
        범위 검색 시 연속 블록 -> 디스크 효율

        테이블당 1개만(한 순서로 정렬)
        중간 삽입 시 재 배치 사용
        ex : PK (my_sql)   

    비클러스터 : 테이블당 여러 개 , 다양한 조회패턴, 데이터는 마구잡이로 집어넣음

        인덱스-> 힙 2단계 접근(랜덤I/o)

    복합 인덱스 : 순서가 중요 (=조건을 앞, 범위 조건 뒤)(=을 뒤에달면 풀스캔 위험!)

    covering index - 힙접근 제거, 중요한 요건에만

        CREATE INDEX idx_cov ON usage_log
        (sub_id, use_dt) INCLUDE (bytes);
        쿼리에 필요한 컬럼을 인덱스가 다 포함(bytes라는 정보)
        힙 접근 없이 인덱스만으로 결과
        INCLUDE로 비검색 컬럼도 탑재
        인덱스 커짐 → 핵심 조회에만
        SELECT use_dt, bytes
        FROM usage_log WHERE sub_id=1001;

    선택도와 인덱스 효율 : 높은 선택도(값이 다양)(결과를 크게 좁힘)(좋음), 낮은 선택도(값이 몇종류 뿐)(많은 행 매칭)(효과 적음)

        자주쓰는 where join orderby 컬럼, 선택도 높은 컬럼, 복합은 등치->범위 순서, 핵심조회는 커버링

쿼리 리라이팅

    옵티마이저가 더 나은 계획을 찾도록 쿼리를 다시 쓴다 - 데이터 결과는 그대로, 표현만 바뀜

        진단(explain 병목확인) -> 변경 -> 측정(explain)
        튜닝의 두 축 : 인덱스, 리라이팅(인덱스를 탈수 있도록 코드 길어지더라도 함수안에 집어넣는거 하지말기)
            왜 인덱스를 못 타나?
            인덱스는 원본 컬럼 값으로 정렬돼 있는데, EXTRACT()· *1.1 로 가공하면 정렬이 깨져
            색인을 못 씀 → 풀스캔.

    원리, 5대 법칙 : SARgable 조건, 필요한 컬럼,행만(명시잘하기)(일찎필터), 적절한 스캔 유도, 불필요 연산 제거(습관적 distinct, order by, 영향없는 서브쿼리,   union(중복이 없는데 사용)), 집합적 사고

        OR → UNION으로 각각 인덱스
            왜? 인덱스는 보통 한 컬럼 기준으로만 타는데, a=1 OR b=2 처럼 다른 컬럼을 OR로
            묶으면 둘 다 못 타 풀스캔되기 쉽다. → 조건을 갈래로 나눠 UNION 으로 합치면 갈래마
            다 자신의 인덱스(a용·b용)를 탄다.

    핵심 코드에 반복문(for/while)으로 쿼리를 반복 호출하는 게 보이면 “집합으로 바꿀 수 있나?” 의심 — 100건이면 1+100=101번이 1번으로. 찾는 법은 EXPLAIN이 아니다
