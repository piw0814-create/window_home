## python 실행 구조·현대 개발환경과 데이터 처리 기초
데이터 분석을 위한 Python 이해 · Day1

## 학습 목표
Python 3.11 실행 흐름과 venv 기반 로컬 실무 환경 구성
컴프리헨션·제너레이터·dataclass로 데이터 레코드를 효율적으로 모델링
Pydantic·mypy·Ruff·pytest로 타입 안전하고 재현 가능한 분석 코드 작성
asyncio·멀티프로세싱으로 데이터 수집·처리 성능 최적화

## 핵심 개념
venv(가상환경) : 프로젝트마다 라이브러리를 따로 담는 독립상자로, 다른 프로젝트와 버전이 충돌하지 않게 해준다

컴프리헨션(comprehension) : for 와 조건을 한 줄로 압축해 리스트,딕셔너리 집합을 만드는 문법

제너레이터(generator) : 값을 한꺼번에 만들지 않고 필요할 떄 하나씩 흘려보내, 대용량 데이터도 적은 메모리로 처리함

dataclass : 데이터 한 건(예 : 주문)을 담는 상자를 몇 줄로 정의해 주는 도구

타입 힌트 : 변수 함수에 어떤 종류의 값이 오가는지 표시해, 실수와 버그를 미리 잡게 돕는 메모

pydantic : 들어온 데이터가 정한 형식(스키마)에 맞는지 자동으로 검증하고 걸러주는 검문소 같은 도구

Ruff : 코드 스타일 오류를 찾아 주고 자동으로 정리해 주는 매우 빠른 린터/포매터
    ruff check
    → 문제 찾기

    ruff format
    → 코드 모양 정리

pytest : 내가 만든 함수가 기대대로 동작하는지 자동으로 확인해 주는 테스트 도구
    assert 실제값 == 기대값

asyncio(비동기) : 여러 api요청을 동시에 처리
    async def: 비동기 함수 정의
    await: 응답을 기다리는 동안 다른 작업에 실행 기회를 넘김
    asyncio.gather(): 여러 비동기 작업을 함께 실행
    asyncio.run(): 비동기 프로그램 시작

## 심화
AST와 PVM
    Python은 문법구조(AST)로 해석한 뒤 바이트코드로 변역하고, 그 바이트 코드를 PVM이 한 줄씩 실행

venv를 쓰는 이유
    프로젝트마다 필요한 라이브러리 버전이 다른데, 컴퓨터 전체에 하나로만 깔면 A프로젝트를 고치다 B가 깨진다.
    venv는 프로젝트별로 독립된 상자를 만들어 이런 충돌을 원천 차단
    requirements.txt와 함께 쓰면 남의 컴퓨터에서도 똑같은 환경을 그대로 재현할 수 있다

type hint와 Pydantic이 버그를 미리 막는 원리
    데이터가 파이프라인을 흐르다 엉뚱한 형식이 섞이면 한참 뒤에 가서야 터진다.
    타입 힌트로 형식을 적어두면 mypy가 실행 전에 문제를 잡아주고, Pydanic은 데이터가 들어오는 입구에서 형식을 검문해 잘못된 값을 초반에 걸러냄

왜 데이터 분석에 파이썬?
    읽기 쉽고, 데이터를 다루는 많은 라이브러리

자료구조를 골라 쓰는 기준
    데이터를 담을 그릇은 여러 종류가 있고, 무엇을 담느냐에 따라 알맞은 그릇이 다르다.
    순서중요, 나중에 값을 바꿈 -> 리스트
    이름으로 값을 찾고싶다 -> 딕셔너리
    한번 정하면 바뀌면 안되는 값 -> 튜플

함수형 도구, 파일 포맷, 예외 - 파이프라인을 짧고 튼튼하게
    데이터 처리는 모든 값에 같은 규칙 적용(map), 조건에 맞는 값만 남기기(filter), 누적해서 하나로 접기(reduce)의 반복
    functools의 partial은 자주 쓰는 함수의 인자 일부를 미리 고정해 이름만 짧게 부르게 해주고, reduce는 목록을 하나의 값으로 접을떄 쓴다.
    이런 함수형 도구를 알면 for문으로 길게 쓰던 코드가 한줄에 정리

    파일 포맷도 골라 쓸 줄 알아야함. CSV는 사람이 열어 보기 좋지만 용량이 크고 타입 정보가 없음. JSON 은 중첩 구조를 담기 좋아 API응답에 흔함. Parquet는 열 단위로 압축 저장해 대용량 분석에서 훨씬 빠르고 적음 - 중간 산출물들은 Parquet로 저장하는게 실무 관행

    경로는 문자열을 '+'로 이어붙이면 os마다 깨지므로, pathlib의 Path로 다뤄 안전하게 만든다

    마지막으로 파일이 없거나 형식이 깨진 데이터는 반드시 생기므로, try/except로 '터지면 멈추는'대신 '잘못된 건만 건너뛰고 로그를 남기는'예외 처리 패턴을 몸에 익혀야 파이프라인이 밤새 안전하게 돔

코드 품질 - Ruff,pytest 디버거가 분석가에게 필요한 이유
    분석 코드는 한번 돌리고 버리는 것 같지만, 실제로는 몇 주 뒤의 내가 다시 열어 고치고 남에게 넘긴다. 그때 품질 도구가 시간을 지켜준다

    Ruff는 린터(문법,스타일 오류를 잡아 주는 검사기)와 포매터(들여쓰기,따옴표를 자동 정리)를 하나로 합친 도구,
        ruff check, ruff format

    pytest는 이 함수는 이런 입력에 이런 값을 내야 한다를 코드로 박아두는 것.
        assert 조건 참 판별

비동기와 병렬 - 기다림과 계산을 나눠 빠르게 
    asyncio의 async/await는 이 '기다리는 시간'을 겹치게 해줌, 한 요청을 보내고 응답을 기다리는 동안 다른 요청을 미리 보내 두는 방식. 네트워크처럼 기다림이 많은 작업에서 수십배 빨라짐.
    httpx 는 이 async 를 지원하는 최신 HTTP 클라이언트라 requests 대신 쓴다.
        async def, await client.get("주소"), await asyncio.gather()

    반대로 무거운 계산(이미지 변환·대용량 파싱 같은 CPU bound)은 기다림이 아니라 진짜 연산이 병목이므로, asyncio 로는 빨라지지 않는다. 이때는 멀티프로세싱으로 CPU 코어 여러 개에 일을 나눠 준다.
        results = pool.map(heavy_calculation, numbers)

    판단 기준: '기다림이 많으면 asyncio, 계산이 많으면 멀티프로세싱'. 그리고 timeit·cProfile 로 실제 어디가 느린지 측정한 뒤에 고른다 - 짐작으로 최적화하지 않는다.

    timeit
    → 작은 코드 조각의 실행 시간을 반복 측정

    cProfile
    → 프로그램 전체에서 어떤 함수가 시간을 많이 쓰는지 분석
