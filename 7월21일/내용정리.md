join의 실제 물리적 알고리즘은 옵티마이저가 결정 : nested loop(한명씩), hash(해시표 만들어 대조), sort_merge(정렬 후 훑기)

nested loop join 

    바깥 테이블의 각 행마다 안쪽 테이블을 탐색해 짝을 찾는다. 안쪽 조인 컬럼에 인덱스가 있으면 빠름
    유리 : 바깥이 작고, 안쪽 조인 컬럼에 인덱스 있을때, 등가, 범위 모두, 소량 조회
    불리 : 두테이블 모두 크고 인덱스 없을떄 -> 행 수의 곱 만큼 느림

hash join 

    유리 : 대용량, 인덱스 없을때, 등가 조인, 배치 분석 
    제약 : 오직 등가(=)조건일 때만, 메모리 초과 시 디스크 스플릿
    build(join 컬럼 기준 해시 테이블, 메모리로 저장) - probe(큰 테이블을 한번 훑으며 해시로 즉시 매칭) 

sort merge join

    유리 : 이미 정렬됨, 정렬이 어차피 필요, 범위 조인 가능, 대용량 정렬 조인
    불리 : 정렬되지 않은 대용량 -> 정렬 비용이 큼
    해시와의 차이 : 해시는 등가만 가능, sort-merge 는 범위 조인도 가능

옵티마이저는 어떻게 물리적 알고리즘을 고르나

    조인조건, 예상 행수, 인덱스 유무, 정렬 상태, 가용 메모리

    카디널리티 : 조건을 통과할 것으로 예상되는 행의 절대 개수, 적으면 nested loop join 많으면 hash 쪽으로 기움
    선택도 : 전체 중 조건을 통과하는 비율(0~1) = 카디널리티 % 전체 행 수, 낮을수록 많이 걸러져 인덱스 조회가 유리

3개 이상 테이블은 합치는 순서가 성능을 좌우한다. 옵티마이저는 여러 순서의 비용을 추정해 가장 싼 순서를 고른다

재귀 CTE 계층 구조 탐색

    앵커(기준 테이블) + 재귀(자기 참조), 종료 조건 작성해둠

    WITH RECURSIVE org AS (
    -- ① 앵커: 시작 행(최상위)
    SELECT id, name, parent_id, 1 AS lvl
    FROM dept WHERE parent_id IS NULL
    UNION ALL -- 결과를 계속 쌓음
    -- ② 재귀: 자식으로 한 단계씩
    SELECT d.id, d.name, d.parent_id, o.lvl+1
    FROM dept d JOIN org o ON d.parent_id = o.id
    ) -- 자식 없으면 자동 종료
    SELECT * FROM org ORDER BY lvl;

트래젝션 : 하나의 논리적 작업 단위 : begin commit rollback

    ACID : 원자성(모든작업이 되야됨), 일관성(트래잭션 전 후로 제약 무결성을 항상 만족하는 상태 유지), 고립성(임시 메모리, 트래잭션이 동시 실행되도 혼자 도는 것처럼 서로 간섭차단), 지속성(커밋된 결과는 정전,크래시에도 보존)

    동시성 이상현상 : 고립성이 약할때 동시에 도는 두 트래잭션 사이에서 생기는 3가지 읽기 사고 -> 격리수준 높이기

        dirty read, non-reapeatable, phantom
        격리 수준 : read uncommitted, read committed, repeatable read, serialzable

    MVCC : 다중 버전 동시성 제어 : 수정 시 기존행을 덮어쓰지말고 새버전을 생성, 각 트래잭션은 자기 시점의 스냅샷을 본다 - vacuu:남아있는 임시메모리 삭제

    락 : 낙관적 vs 비관적 : 두방식 모두 갱신손실 : 두 트래잭션이 같은 행을 읽고 각자 덮어써 한쪽 변경이 사라지는 문제를 막는 장치

        비관적 락 : 충돌가정 -> 미리 잠그고 시작
        낙관적 락 : 잠그지 않고 일단 수정 -> 저장 직전 남이 바꿨나를 version번호로 확인

        advisory lock(권고 잠금) :  개발자가 약속한 숫자를 잠금 -행과 무관
            row lock (행 락)
        
        데드락 : 서로를 기다리는 교착 -> 한쪽을 강제 롤백 -> 앱은 재시도 : 완전히 없앨 수 없음, 발생빈도를 줄이고 재시도 로직을 갖추는 것이 현실적 대응

            자원을 같은 수서로 잠금, 트래잭션을 짧게 유지, 필요한 행만 좁게 잠금, 타임아웃+재시도
